How to use this roadmap

Every week has one clear target — a topic to learn and something small to build with it. Spend 45–90 focused minutes a day, 5–6 days a week. In a hard exam week, do less — never zero. Every resource is linked directly below its week, and the full library is collected near the bottom if you'd rather browse by tool.

  • Push whatever you build to GitHub the same week you build it — don't let a backlog form.
  • One language, one framework, one new tool at a time. Depth beats breadth for the first year.
  • If you already know a topic in a given week, skip ahead — this is a floor, not a cage.
Q1

Weeks 1–13 — Python from zero, real fundamentals

Goal: go from never having written code to being able to open a blank file and build something, without panicking. Everything else in this roadmap depends on this quarter being solid.

WkTarget
1Install Python + VS Code. Variables, data types, input/output, if/else. Write 5 tiny programs (temperature converter, even/odd checker, simple calculator).
CS50P
2Loops (for/while), basic string manipulation. Build: a number-guessing game, a text countdown timer.
CS50P — Week 2–3
3Functions — parameters, return values, scope. Rebuild Week 1–2 programs using functions instead of one long script.
4Lists and tuples — indexing, slicing, common methods. Build: a contact list stored as a list of dictionaries.
5Dictionaries and sets. Build: a word-frequency counter that reads a text file.
6File I/O (reading/writing files), error handling with try/except. Build: a CLI to-do list that saves to a file.
7OOP part 1 — classes, objects, __init__, methods, attributes. Rebuild the to-do list using a Task class.
8OOP part 2 — inheritance, encapsulation, polymorphism. This is the concept set that transfers directly to every other language later.
9Git & GitHub properly: init, add/commit/push, branches, .gitignore, real commit messages. Push everything from Weeks 1–8 into one organized repo.
Odin Project — Git lessons
10LeetCode Easy — arrays & strings. Target: 4–5 problems. Write the brute-force solution first, then optimize.
NeetCode roadmap LeetCode
11LeetCode Easy — hashmaps & two pointers. Target: 4–5 problems.
12Recursion basics, plus an intuitive (not mathematical) feel for Big-O: what makes something O(n) vs O(n²).
13Catch-up + consolidation. Finish anything unfinished. Push a clean, well-documented Python repo to GitHub.
Checkpoint: write a working Python program from a blank file without googling syntax every two minutes, explain OOP in plain English, ~10–15 LeetCode Easy problems done.
Q2

Weeks 14–26 — SQL depth, JavaScript basics, C# revisit

Databases and the browser enter the picture. C# comes back not as a new language, but as "same concepts, new syntax."

WkTarget
14SQL refresher: SELECT, WHERE, ORDER BY, basic filtering.
SQLZoo
15SQL joins — INNER, LEFT, RIGHT — until you can explain each without hesitating.
SQLZoo — Joins
16SQL aggregates — GROUP BY, HAVING, COUNT/SUM/AVG. Practice against any database you can access or build a small sample one.
17SQL subqueries vs CTEs, and normalization basics (1NF–3NF) — conceptual, not memorized.
18JavaScript basics: variables, functions, arrays, objects. Same concepts as Python, new syntax — this should move fast.
Odin Project — Foundations MDN JavaScript
19DOM manipulation basics (querySelector, event listeners). Build: an interactive to-do list in the browser (HTML+CSS+JS).
20Fetch API, async/await basics — enough to call a public API (e.g. a weather API) and display the results on a page.
21Back to DSA — linked lists. Target: 4 problems in Python.
LeetCode
22DSA — stacks & queues. Target: 4 problems.
23C# revisit begins — same OOP concepts, new syntax. Rebuild one small Python program (e.g. the to-do CLI) in C#.
Microsoft — Get Started with C# C# interactive tutorials
24Keep building small C# programs until the syntax stops feeling foreign.
25Start ASP.NET Core basics — what a Web API is, routes, controllers. Follow a tutorial and build a "hello world" API.
ASP.NET Core — first Web API
26Catch-up + consolidation. Push SQL practice queries and the JS to-do app to GitHub with proper READMEs.
Checkpoint: comfortable writing real SQL queries against your own schema, a working browser-based JS app, and C# feels familiar again.
Q3

Weeks 27–39 — Backend project, React, start applying

This is the quarter where the pieces connect into a real, deployed full-stack app — and where applications start going out.

WkTarget
27Connect an ASP.NET Core Web API to a database via Entity Framework Core — basic CRUD on one table.
ASP.NET Core tutorial
28Finish a full CRUD API — e.g. a "Client Tracker." Test every endpoint with Postman.
Postman
29Add basic authentication (JWT) to the API — register/login endpoints.
30React basics: components, JSX, props. Rebuild a static page as a set of React components.
react.dev — Learn React
31React useState, handling forms and events. Build: a form that adds items to an on-screen list.
32React useEffect, fetching data from your own API (the backend from Weeks 27–29).
33Connect frontend and backend fully — React talking to your API, auth flow working end-to-end.
34DSA — trees (inorder/preorder/postorder traversal, BFS/DFS). Target: 5 problems.
NeetCode
35DSA — graph basics (BFS/DFS on graphs). Target: 4 problems.
36Update resume and LinkedIn with the full-stack project. Start applying to internships this week — don't wait to "feel ready."
37Behavioral interview prep — STAR method. Prepare 5 real stories from your own projects and experience.
38Contribute to one open-source repo — find a "good first issue" tag, even something small.
39Catch-up + polish. Deploy your full-stack app live so it's a real link, not just code.
Checkpoint: one deployed full-stack project (frontend + API + database + auth), applications going out, and you can talk through your projects fluently.

Non-negotiables

  1. One language at a time. Trying to keep four languages equally warm from zero is how people end up shallow in all of them.
  2. DSA never fully stops once it starts (Week 10). Two problems in a brutal week is enough to keep the habit alive — don't let it drop to zero.
  3. Every project goes on GitHub the same week it's built. A backlog of "I'll push it later" always wins if you let it start.
  4. Don't add a new tool or framework until the current one has a working project behind it — not just a finished tutorial.
  5. Applying starts in Week 36, not when you "feel ready." Readiness is proven in interviews, not decided in advance.
